## Primary Objective

Build a localhost web application providing a user-friendly interface for the CEO Personal OS framework. The dashboard replaces manual file editing with forms and visualizations while keeping markdown files as the single source of truth.

### Key Requirements

1. **Daily Review Form** - Create/edit daily check-ins via web form
2. **Life Map Visualization** - Radar chart showing 6 life domains
3. **Recent Reviews List** - Browse and view past reviews
4. **Direct Markdown R/W** - No database, read/write markdown files directly

### Success Metrics

- User completes daily review in <5 minutes
- Dashboard loads in <2 seconds
- Markdown files remain human-readable
- No data loss during save/edit operations

```typescript

  || path.resolve(process.cwd());

```

## Technical Notes

### shadcn/ui Integration

shadcn/ui is NOT a traditional npm package. It's a CLI tool that copies component source code into your project:

1. **Installation:** `npx shadcn@latest init` creates `components.json` and `lib/utils.ts`
2. **Adding components:** `npx shadcn@latest add button` copies Button component to `src/components/ui/button.tsx`
3. **Customization:** Edit the copied files directly - you own the code
4. **Styling:** Uses Tailwind CSS with CSS variables for theming
5. **Dependencies:** Each component may install its own Radix UI primitives

### Tailwind CSS with Next.js 14

1. **Configuration:** `tailwind.config.js` with content paths to `src/**/*.{tsx,ts}`
2. **CSS Variables:** shadcn uses HSL colors via CSS variables for theming
3. **Dark mode:** Configured via `darkMode: ["class"]` (not implemented in MVP)
4. **PostCSS:** Standard `postcss.config.js` with tailwindcss and autoprefixer

### File System Access

Next.js App Router API routes run on the server and can access the file system via Node.js `fs` module. Key considerations:

1. **Absolute paths:** Use `process.env.MARKDOWN_BASE_PATH` for the project root
2. **Error handling:** Wrap all fs operations in try/catch
3. **Atomic writes:** Write to temp file, then rename to avoid data loss
4. **Encoding:** Always use `utf-8` encoding for markdown files

## Risk Mitigation

| Risk | Impact | Mitigation |
|------|--------|------------|
| Markdown parsing fails | Data loss | Best-effort parsing, show raw content on failure |
| File system errors | User frustration | Clear error messages, retry logic |
| Concurrent edits | Data loss | Last-write-wins (acceptable for single-user) |
| Large file reads | Slow performance | Lazy loading, pagination for reviews list |
